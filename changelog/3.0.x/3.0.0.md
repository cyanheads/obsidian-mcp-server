# 3.0.0 — 2026-04-27

Major version: complete rewrite of `obsidian-mcp-server` on top of [`@cyanheads/mcp-ts-core`](https://www.npmjs.com/package/@cyanheads/mcp-ts-core). The server now exposes the Obsidian Local REST API plugin as a typed, declarative MCP surface — 14 tools and 3 resources, with section-aware editing, three-mode search, and tag reconciliation across both inline and frontmatter representations. The 2.x tool surface is gone; consumers must update their integrations.

## Migrating from 2.x

The 2.x tool surface is gone. Every tool name and input/output schema changed — consumers must update their integrations.

### Tool name map

| 2.x | 3.0.0 |
|:----|:------|
| `obsidian_read_note` | `obsidian_get_note` (`format`: `content` / `full` / `document-map` / `section` — replaces the v2 `markdown` vs `json` toggle and adds section-projection reads) |
| `obsidian_update_note` (single tool, `append` / `prepend` / `overwrite` modes) | Split by mode into `obsidian_write_note` (full-file PUT or in-place section replace), `obsidian_append_to_note`, and `obsidian_patch_note` (append / prepend / replace against headings, block refs, frontmatter fields — section-aware edits weren't possible in v2) |
| `obsidian_search_replace` | `obsidian_replace_in_note` (now honors `$1` / `$&` capture-group references in regex replacements) |
| `obsidian_global_search` | `obsidian_search_notes` — three modes: text, Dataview DQL, JSONLogic. The v2 paginated result shape is gone; v3 caps at 100 hits with an `excluded` indicator and a narrowing hint. |
| `obsidian_list_notes` | `obsidian_list_files` (same extension and name-regex filters; result echoes `appliedFilters`) |
| `obsidian_manage_frontmatter` | Same name. Atomic `get` / `set` / `delete` on a single key. |
| `obsidian_manage_tags` | Same name. Now reconciles inline `#tag` and frontmatter `tags:` together; skips fenced code blocks. |
| `obsidian_delete_note` | Same name. Uses `ctx.elicit` for human-in-the-loop confirmation when the client supports it. v2's silent case-insensitive path fallback was dropped — delete now requires an exact path match so a typo can't quietly delete a different file on case-sensitive filesystems. |

`obsidian_list_tags`, `obsidian_list_commands`, `obsidian_open_in_ui`, `obsidian_patch_note`, and `obsidian_execute_command` (opt-in via `OBSIDIAN_ENABLE_COMMANDS=true`) are new in 3.0.0. v2 had no resources; 3.0.0 adds three: `obsidian://vault/{+path}` (note as parsed `NoteJson`), `obsidian://tags`, and `obsidian://status`.

### Removed

- **`VaultCacheService`** and its env vars (`OBSIDIAN_ENABLE_CACHE`, `OBSIDIAN_CACHE_REFRESH_INTERVAL_MIN`). The in-memory vault cache and its background refresh loop are gone — the server is stateless. Search and list operations hit the Local REST API on every call. If you relied on the cache as a search fallback when the upstream was unreachable, that fallback no longer exists; failures now surface as service errors. For large vaults, narrow searches with the text-mode `pathPrefix` filter or use Dataview/JSONLogic predicates against `file.path` / `path`.

### Default and config changes

- `OBSIDIAN_VERIFY_SSL` default changed from `true` to `false`. Both plugin endpoints (HTTP on 27123, HTTPS on 27124 with a self-signed cert) work better with verification off; the v2 default of `true` required explicit overrides for almost every realistic config.
- `OBSIDIAN_BASE_URL` default unchanged from v2 (`http://127.0.0.1:27123`); no longer marked required because the Zod default supplies it.
- New runtime requirement: `engines.bun` ≥ 1.3.11. Node ≥ 22 still works as a runtime; the install-time constraint is Bun-pinned because the dev workflow (build, test, devcheck) assumes it.

### Architectural notes

- The server is no longer a fork of `cyanheads/mcp-ts-template`; it depends on [`@cyanheads/mcp-ts-core`](https://www.npmjs.com/package/@cyanheads/mcp-ts-core) as a package. Tools and resources are declarative definitions registered via `tool('name', { ... })` and `resource('uri', { ... })` builders.
- Per-request cancellation: `ctx.signal` propagates through to the upstream HTTP client, so client cancellations and per-request timeouts cut off in-flight Local REST API calls.
- HTTP-mode auth (`MCP_AUTH_MODE=jwt|oauth`) and tenant-aware logging via `ctx.tenantId` are framework features, retained from v2's posture.

## Added

- **Read tools:** `obsidian_get_note` (content / full / document-map / section projections), `obsidian_list_files` (with extension and name-regex filters), `obsidian_list_tags`, `obsidian_list_commands`, `obsidian_search_notes` (text / Dataview DQL / JSONLogic).
- **Write tools:** `obsidian_write_note` (full-file PUT or in-place section replace), `obsidian_append_to_note`, `obsidian_patch_note` (append / prepend / replace against headings, block refs, frontmatter fields), `obsidian_replace_in_note` (string and regex search-replace).
- **Manage tools:** `obsidian_manage_frontmatter` (atomic get / set / delete on a single key), `obsidian_manage_tags` (reconciles frontmatter `tags:` and inline `#tag`, skipping fenced code blocks).
- **Destructive tools:** `obsidian_delete_note` (with `ctx.elicit` confirmation when supported), `obsidian_open_in_ui`.
- **Opt-in tool:** `obsidian_execute_command` — registered only when `OBSIDIAN_ENABLE_COMMANDS=true`.
- **Resources:** `obsidian://vault/{+path}` (note as parsed NoteJson), `obsidian://tags`, `obsidian://status`.
- **Service layer:** Typed Local REST API client with `ctx.signal` propagation, frontmatter parse/serialize/edit helpers, section extraction across headings/blocks/frontmatter.
- **Configuration:** Five `OBSIDIAN_*` env vars wired through `parseEnvConfig` with Zod validation.
- **Search guardrails:** 100-hit cap with an `excluded` indicator that surfaces overflow count and a narrowing hint.
- **Deployment:** Dockerfile (HTTP transport, stateless session, OCI metadata) and `server.json` manifest for MCP registry listings.
- **Path resolution UX:** Read and open tools (`obsidian_get_note`, `obsidian_open_in_ui`) share a `withCaseFallback()` helper. On a 404 against a `path:` target, the helper lists the parent directory and (a) silently retries against the canonical filename if exactly one case-insensitive match exists — fixing `Readme.md` vs `README.md` on case-sensitive filesystems — (b) throws `Conflict` with the candidates when multiple case matches exist, or (c) re-throws `NotFound` enriched with `data.suggestions[]` and a `Did you mean: "X", "Y"?` hint when only extension-stripped near-matches exist. Resolved canonical paths flow back through `result.path` so callers see what was actually touched. `obsidian_delete_note` deliberately opts out — silent path normalization on a destructive op is too dangerous when not every client supports `ctx.elicit` confirmation.

## Changed

- `obsidian_write_note` now sets `Apply-If-Content-Preexists: true` on section-targeted PATCH and strips a leading duplicate heading line so a caller's `## Section` does not get embedded inside the section body.
- `obsidian://status.authenticated` is now a separate authenticated probe against `/vault/` instead of the unauthenticated `/` field — the resource still serves reachability info when the API key is wrong.
- Service errors report the vault-relative path (e.g. `Not found: Notes/foo.md`) instead of the internal HTTP path (`/vault/Notes/foo.md`). Periodic-note paths render as `daily note for 2026-04-26`, so 404s and log lines name the resource the caller asked for instead of the route.
- 400 responses that look like a missing PATCH section target are translated into actionable guidance pointing at `obsidian_get_note` `format: "document-map"`, with a reminder that nested headings need `Parent::Child` syntax.
- `obsidian_get_note` `format()` renders ISO 8601 timestamps for `stat.ctime` / `stat.mtime` instead of raw epoch ms.
- `obsidian_search_notes` empty-result rendering includes a narrowing hint; the JSONLogic mode description now enumerates the available `var` paths (`path`, `content`, `frontmatter.<key>`, `tags`, `stat.{ctime,mtime,size}`).
- `obsidian_list_files` echoes any active filters as `appliedFilters` (extension and/or `nameRegex`) in both `structuredContent` and the rendered text twin, so callers can audit which filters narrowed the listing.
- `obsidian_open_in_ui` no longer carries `destructiveHint` — the `failIfMissing: true` default keeps the call read-like; opt-in create-on-open lives behind that flag. `obsidian_append_to_note` is now correctly flagged `destructiveHint`.
- `obsidian_write_note` and `obsidian_append_to_note` outputs drop the `updatedContent` echo. The Local REST API endpoints don't reliably return a body for the whole-file path, so the field was effectively a coin flip; section-targeted edits already round-trip the new content through a separate read.
- Tool descriptions across `obsidian_*` rewritten to drop internal terms (`OBSIDIAN_ENABLE_COMMANDS`, "upstream", "PATCH-with-append", "404", "NoteJson", "Echoed") in favor of user-facing concepts and concrete next steps. `obsidian_get_note` now documents that heading-section projections include the full subtree under that heading; `obsidian_list_files` clarifies that `extension` filters files only (directories pass through); `obsidian_manage_tags` documents inline-location additions appending at end of file and `list` ignoring the `tags` field; `obsidian_append_to_note` documents block-reference concatenation (no separator inserted by upstream); `ContentTypeSchema` now shows the JSON-literal encoding requirement so `contentType: "json"` callers don't pass bare scalars and hit an opaque upstream 400; `obsidian_open_in_ui` and `obsidian_manage_frontmatter` had top-level cross-tool routing hints removed.
- `obsidian_search_notes` annotations include `idempotentHint: true` alongside `readOnlyHint: true` — same query → same results, both hints apply.
- Tightened `engines.bun` from `>=1.2.0` to `>=1.3.11` so it matches the pinned `packageManager` and the documented prerequisite. Older Bun runtimes no longer satisfy the install constraint.
- `obsidian_manage_frontmatter` `delete` no longer refetches the note after the write — it projects the post-state frontmatter locally from the prior read by stripping the deleted key. One fewer round-trip per delete; the upstream PUT is the source of truth either way.
- `obsidian_manage_tags` short-circuits on no-op reconciles. When `add`/`remove` resolves to zero applied changes, the handler skips both the upstream PUT and the post-write GET refetch and returns directly from the pre-read note. Re-adding tags a note already carries now costs one round-trip instead of two.
- `obsidian_open_in_ui` skips the existence probe when `failIfMissing: false`. The probe only existed to populate `createdIfMissing`; under the create-on-open contract the answer is unambiguously `true`, so the open call is now a single round-trip with no upfront `GET /vault/<path>` or parent listing.
- `obsidian_search_notes` rejects `pathPrefix` outside `text` mode with `InvalidParams`. The filter was silently ignored in `dataview` and `jsonlogic` (those modes already filter by `path` / `file.path` predicates), which made result counts surprising. Callers using `pathPrefix` in those modes now get an actionable error pointing at the right surface.
- `npm start` script added (`node dist/index.js`) for distribution-time launches that don't go through the `MCP_TRANSPORT_TYPE`-prefixed `start:stdio` / `start:http` variants.
- Dependency bumps: `@cyanheads/mcp-ts-core` 0.7.5 → 0.7.6, `@biomejs/biome` 2.4.7 → 2.4.13, `typescript` 5.9.3 → 6.0.3 (major), `vitest` 4.1.0 → 4.1.5.
- README polish: `obsidian_open_in_ui` row now documents both `failIfMissing` and `newLeaf` toggles; the project-structure table no longer references `docs/design.md` (the design notes have been removed — they were superseded by the shipped implementation).
- Internal cleanup: added `ObsidianService.resolvePath()` to collapse the repeated `target.type === 'path' ? … : getNoteJson()` ternary across handlers; `splice()` in `frontmatter-ops` now captures YAML directly from the regex match group; `obsidian_open_in_ui` consolidates two near-duplicate try blocks into one. Document-map fetch and path resolution run in parallel for non-`path` targets.
- Test infrastructure: `ObsidianService` constructor now accepts an optional `fetchImpl` (`(url, init) => Promise<Response>`) instead of an undici `Dispatcher`. Tests inject a stub fetch directly because Bun's runtime treats `undici` as a builtin and silently ignores `vi.mock('undici', …)` — so module-level mocking is a dead end under `bunx vitest run`. The harness exposes the same `pool.intercept(matcher).reply(...)` surface, no test bodies changed.

## Fixed

- `obsidian_replace_in_note` now honors `$1` / `$2` / `$&` capture-group references in regex replacements (the regex path now uses the string overload of `String.replace` instead of the callback overload, which silently drops capture-group templates).
- `obsidian_list_tags` and `obsidian://tags` return `{name, count}` matching the upstream `/tags/` shape (was `{tag, count}`).
- `NoteTarget.date` aligned with the Zod-inferred shape (`string | undefined`) so handlers no longer need an `as NoteTarget` cast under `exactOptionalPropertyTypes`.
- `OBSIDIAN_VERIFY_SSL=false` actually disables TLS verification under Bun's runtime now. Bun ignores undici's per-dispatcher `connect.rejectUnauthorized`, so calls to the HTTPS port (`https://127.0.0.1:27124`, self-signed) failed with `DEPTH_ZERO_SELF_SIGNED_CERT` regardless of the env var. The service constructor sets `NODE_TLS_REJECT_UNAUTHORIZED=0` when `verifySsl` is false **and the runtime is Bun** — Node honors the dispatcher option directly, so the env-var fallback is scoped to Bun and Node deployments are not mutated process-wide.
- `extractSection` (heading and block) now skips a leading YAML frontmatter block before scanning. Without this guard, a `# yaml comment` line inside frontmatter would false-match as a Markdown heading, and block extraction's paragraph walk-back could pull the closing `---` fence into the result. Both extractors now start at the first body line.
- `obsidian_manage_frontmatter` `get` on an absent key now returns `value: null` in `structuredContent` instead of omitting the field. Closes a parity gap — `content[]` rendered `_(absent)_` while structured callers got `undefined` and a missing key.
- `ObsidianService.probeAuthenticated` re-throws when `ctx.signal` is aborted instead of catching the abort and reporting `false`. A cancelled or timed-out probe was masquerading as a 401 in `obsidian://status.authenticated`; abort errors now propagate so cancellation surfaces as a service error rather than a silent `authenticated: false`.

## Security

- Upstream error bodies are normalized through a `safeUpstream()` helper before being attached to JSON-RPC `error.data` — the helper preserves only the human `message` field (or trims raw text to 200 chars) and drops plugin-internal fields like `errorCode` that would otherwise leak into client-visible error payloads.
- The TLS verify-disable fallback (`NODE_TLS_REJECT_UNAUTHORIZED=0` when `verifySsl: false`) is **scoped to Bun**. The server makes outbound HTTPS only to the configured Obsidian Local REST API, so the blast radius is the upstream you already chose to trust by setting `verifySsl: false`. Node deployments rely on undici's per-dispatcher `rejectUnauthorized: false` and don't mutate process-wide TLS behavior. The `OBSIDIAN_VERIFY_SSL` env var description calls this out.
